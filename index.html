Introduction: The Pattern Hidden in Plain Sight

Last week, I was in a quarterly planning meeting with stakeholders across three time zones—Amsterdam just starting their day, Sydney winding down, California barely awake. We were reviewing our data governance program's velocity trends, debating whether our current sprint structure was optimal.

"What if we tried two-week sprints instead of three?" someone suggested.

"We could A/B test it," I replied, half-joking.

But here's the thing: we should absolutely A/B test it. And we should automate that testing. And we should let an AI agent iterate on the optimal sprint structure while we focus on strategy.

That realization hit me after reading Andrej Karpathy's work on autoresearch. For those unfamiliar, Karpathy—co-founder of OpenAI, former Tesla AI lead—developed a pattern for autonomous machine learning research. The setup is elegant: you write a program.md file that defines your goal, your metric, and what the agent is allowed to change. Then the agent runs in a loop forever: propose change → test → measure → keep or revert.

It's evolutionary. Only improvements survive. The human sets the strategy; the agent handles execution.

Everyone talks about autoresearch in the context of ML training loops. But here's what I realized: a program manager's entire job is already an autoresearch loop. We just run it manually, slowly, and with way too many meetings.

"A program manager's entire job is already an autoresearch loop. We just run it manually, slowly, and with way too many meetings."

The PM's Daily Loop IS AutoResearch (They Just Don't Know It)

Think about what you do every sprint:

  • Define the goal (sprint objectives, OKRs)
  • Choose an approach (team allocation, process, tooling)
  • Execute (daily standups, progress tracking)
  • Measure outcome (velocity, defect rate, stakeholder satisfaction)
  • Adjust (retrospective, process changes)

This is literally the autoresearch pattern. Standup → sprint → retro → adjust. We've been doing autoresearch since before it had a name.

The difference? We're the ones running the loop. Manually. In my case, I'm coordinating changes across time zones, navigating stakeholder politics, and trying to remember what we decided in last quarter's retrospective. It's slow. It's error-prone. And honestly, most of it is measurable and automatable.

Here's the insight that changes everything: we can automate the parts of program management that are metric-driven iteration.

Not the strategy. Not the stakeholder relationships. Not the judgment calls that require human wisdom. But the execution loops? The testing of incremental process improvements? The iteration on templates, formats, and approaches? That's pure autoresearch territory.

Six Concrete Use Cases

Let me get concrete. Here are six areas where I could deploy autoresearch loops tomorrow if I had the tooling built. These aren't theoretical—these are real pain points from managing cross-timezone data programs at a global trading firm.

1. Stakeholder Communication Optimization

Every Friday, I send status updates to three stakeholder tiers: exec leadership, project sponsors, and working-level teams. The emails are different for each group. But here's what I don't know: which format gets the best engagement?

Should the exec summary be three bullets or five? Should I lead with risks or wins? Should I use RAG status or narrative format? I've been guessing based on experience. But this is measurable.

stakeholder-comms.md
# Goal
Maximize stakeholder engagement with weekly status updates

# Metric
engagement_score = (open_rate * 0.3) + (response_rate * 0.5) + (1 / avg_response_time_hours * 0.2)

# What the agent can change
- Email subject line templates
- Executive summary length (3-7 bullets)
- Ordering of sections (risks-first vs wins-first)
- Level of detail in each section
- Use of data visualizations vs text
- Timing (Friday 4pm vs Monday 9am)

# Constraints
- Must include all required sections (summary, progress, risks, next steps)
- Must maintain professional tone
- Must not exceed 800 words for exec tier
- Each variation tested with minimum 4-week sample

# Evaluation
- Track opens, responses, time-to-response per stakeholder tier
- Segment by tier (exec, sponsor, working-level)
- Keep changes that improve engagement_score by >5%
- Revert if engagement drops by >3%

The agent runs this loop every week. It proposes a variation, sends it to a test segment, measures engagement, and keeps what works. Over a quarter, it learns exactly what communication style resonates with each stakeholder tier.

I'm not guessing anymore. I'm measuring.

2. Meeting Effectiveness

I run a weekly cross-timezone sync: 6 AM in California, 3 PM in Amsterdam, 11 PM in Sydney. Nobody's happy. The meeting often runs over. Action items sometimes slip through the cracks.

Could an autoresearch loop make this better? Absolutely.

meeting-optimization.md
# Goal
Minimize meeting duration while maximizing action item completion

# Metric
effectiveness_score = (1 / avg_meeting_duration_minutes) * action_completion_rate

# What the agent can change
- Agenda template structure
- Pre-read format and length
- Time-boxing per section
- Follow-up format (Slack vs email vs Jira)
- Decision documentation approach

# Constraints
- Meeting duration must be 30-60 minutes
- All critical topics must be covered
- Action completion rate must stay above 85%

# Evaluation
- Measure actual duration vs scheduled duration
- Track action item completion rate (7-day window)
- Survey participants on meeting effectiveness (monthly)
- Keep changes that improve effectiveness_score by >8%

The agent iterates on meeting structure. Does a structured pre-read reduce meeting time? Do time-boxed sections improve focus? Does immediate Slack follow-up boost action completion?

After a quarter, you have data. Not opinions. Data.

3. Risk Register Optimization

Here's an uncomfortable truth: most risk registers are theater. We log risks because governance requires it, but how often do our "high-severity" risks actually materialize? And how often do we miss risks we should have flagged?

This is a prediction accuracy problem. And prediction accuracy is measurable.

risk-prediction.md
# Goal
Improve risk prediction accuracy (% of flagged risks that actually materialize)

# Metric
precision = true_positives / (true_positives + false_positives)
recall = true_positives / (true_positives + false_negatives)
f1_score = 2 * (precision * recall) / (precision + recall)

# What the agent can change
- Risk scoring weights (probability vs impact)
- Categorization criteria
- Escalation thresholds
- Review frequency
- Risk owner assignment heuristics

# Constraints
- Cannot remove human review of high-severity risks
- Must maintain audit trail
- Changes evaluated over rolling 90-day window

# Evaluation
- Track which risks materialize vs which don't
- Measure false positive rate (flagged but didn't happen)
- Measure false negative rate (happened but wasn't flagged)
- Optimize for F1 score (balance precision and recall)

Over time, the agent learns which risk patterns actually lead to issues. Maybe technical risks in the first sprint of a quarter have higher materialization rates. Maybe risks flagged by certain team members are more predictive. The agent figures this out through iteration.

Your risk register becomes a predictive tool, not a compliance checkbox.

4. Sprint Velocity Optimization

Every sprint planning, we debate story sizing. "Is this a 5 or an 8?" We argue about whether to split stories. We wonder if our current team allocation is optimal.

All of this is—you guessed it—measurable.

velocity-optimization.md
# Goal
Maximize story points delivered per sprint while keeping defect rate low

# Metric
velocity_score = story_points_completed * (1 - defect_rate) * (1 - spillover_rate)

# What the agent can change
- Story sizing heuristics
- Team allocation patterns (specialist vs generalist)
- WIP limits
- Story split recommendations
- Sprint length (1-week, 2-week, 3-week experiments)

# Constraints
- Defect rate must stay below 3%
- Team member utilization must stay between 70-90%
- No more than 20% spillover per sprint

# Evaluation
- Track actual velocity vs planned velocity
- Measure defect rate (bugs found in production within 30 days)
- Track spillover rate (stories pushed to next sprint)
- Run A/B tests on sprint structure across parallel teams

The agent experiments with sprint structure. Two-week sprints vs three-week. Stricter WIP limits vs looser. It measures outcomes and optimizes for sustainable velocity (high points, low defects, minimal spillover).

5. Status Report Quality

In my experience managing cross-timezone data programs, I've learned that exec engagement with status reports is a leading indicator of program health. If leadership stops asking follow-up questions, you're in trouble—either they don't understand what you're saying, or they don't care.

Both are fixable through iteration.

status-report-optimization.md
# Goal
Maximize executive engagement with weekly status reports

# Metric
engagement_score = (exec_response_rate * 0.4) + (follow_up_question_count * 0.4) + (forward_rate * 0.2)

# What the agent can change
- Report format (narrative vs structured vs dashboard)
- RAG status criteria and thresholds
- Executive summary style (bullets vs paragraph)
- Data visualization choices (charts vs tables vs text)
- Section ordering and emphasis

# Constraints
- Must fit on 2 pages maximum
- Must include all governance-required sections
- Must be readable in under 5 minutes

# Evaluation
- Track exec response rate to status emails
- Count follow-up questions (more questions = more engagement)
- Measure forward rate (execs sharing with their teams)
- Survey quarterly on report usefulness

More follow-up questions means they're engaged and thinking strategically. The agent learns what format drives that engagement.

6. Vendor/Procurement Optimization

Every PM deals with procurement. RFPs, vendor evaluation, contract negotiation. It's slow. It's painful. And most organizations do it the same way every time, regardless of whether it's working.

But cycle time and quality are both measurable.

procurement-optimization.md
# Goal
Reduce procurement cycle time while maintaining vendor quality scores

# Metric
efficiency_score = (1 / cycle_time_days) * vendor_quality_score * (1 - cost_overrun_rate)

# What the agent can change
- RFP template structure and length
- Evaluation criteria weights
- Negotiation approach (collaborative vs competitive)
- Vendor shortlist size (3 vs 5 vs 7 vendors)
- Decision documentation format

# Constraints
- Must include all compliance-required sections
- Must have minimum 2 vendors in final evaluation
- Vendor quality score must stay above 7/10

# Evaluation
- Measure time from RFP to signed contract
- Track vendor quality scores (performance reviews at 90 days)
- Monitor cost overruns vs initial estimates
- Segment by procurement size and complexity

Maybe shorter RFPs get faster responses without sacrificing quality. Maybe collaborative negotiation works better for strategic vendors. The agent figures it out through experimentation.

Programming the Program: The Meta-Insight

Karpathy talks about "programming the program.md"—the human sets the strategy, the agent handles execution. This is the fundamental shift for program managers.

Traditionally, our job is: "I decide the best approach based on experience, then execute it."

With autoresearch, it becomes: "I define what to optimize and how to measure it. The agent finds the optimal approach through evidence."

This is profound. You're no longer the one deciding whether exec summaries should be three bullets or five. You're the one deciding that executive engagement is the metric that matters. The agent figures out the optimal format through experimentation.

Your job shifts from execution to architecture. From "how do I do this?" to "what should I optimize?"

In my case, managing data programs across three continents, this would be transformative. I spend 60% of my time on execution (writing status reports, running meetings, updating trackers) and 40% on strategy (stakeholder management, roadmap decisions, risk assessment). With autoresearch loops handling the execution iteration, I could invert that ratio.

Imagine spending 60% of your time on strategy, relationships, and judgment calls—the parts that actually require human wisdom—while the agent optimizes the measurable processes in the background.

That's the vision.

Why PMs Are Uniquely Positioned

Here's what surprised me: program managers are uniquely well-positioned to adopt autoresearch. Better than most software engineers. Better than most researchers. Here's why:

1. We already think in metrics. KPIs, OKRs, velocity, burn rate, cycle time—this is our native language. We don't need to learn how to measure outcomes. We already do it.

2. We already run retrospectives. Every sprint, every quarter, we do the "evaluation" step. We ask "what worked, what didn't, what should we change?" We just don't automate the iteration based on those learnings.

3. We already iterate. Sprint cycles, quarterly planning adjustments, continuous improvement—iteration is baked into how we work. We're not trying to do something fundamentally new. We're automating something we already do manually.

4. We have access to the data. Email open rates, meeting durations, action item completion, velocity trends, stakeholder feedback—we already collect most of this. We just don't use it systematically for optimization.

5. Our domain is well-scoped. Unlike "solve AGI" or "optimize the entire business," PM work has clear boundaries. We're optimizing processes, not reinventing them. The scope of each autoresearch loop is manageable.

The tooling doesn't exist yet (though I'm building it—more on that later). But the mental model is already there. If you're a PM reading this, you already understand autoresearch. You just haven't automated it yet.

The Three-File Pattern for PM Work

Karpathy's autoresearch pattern uses three files:

  • prepare.py — the fixed infrastructure (dataset, environment, dependencies)
  • train.py — the thing being iterated on (model architecture, hyperparameters)
  • program.md — the strategy doc (goal, metric, constraints, what can change)

For PM work, this maps beautifully:

prepare.py = Fixed Infrastructure

Your fixed infrastructure is everything that shouldn't change during iteration:

  • Your JIRA/Linear/tooling setup
  • Slack channels and notification patterns
  • Reporting cadence (weekly, bi-weekly, monthly)
  • Team structure and roles
  • Governance requirements and compliance needs

This is your stable platform. The agent doesn't touch this.

train.py = The Thing Being Iterated On

This is the process, template, or approach you're optimizing:

  • The status report template
  • The meeting agenda format
  • The risk scoring heuristic
  • The sprint planning approach
  • The stakeholder communication style

The agent proposes changes here, measures outcomes, keeps what works.

program.md = Your Strategy Doc

This is where you define:

  • Goal: What you're trying to achieve
  • Metric: How you measure success (quantitatively)
  • What can change: The parameters the agent is allowed to modify
  • Constraints: The guardrails (governance, tone, scope)
  • Evaluation: How to decide if a change is an improvement

This is your control surface. You're programming the optimization loop.

In practice, for each PM process you want to optimize, you create these three files. The agent reads program.md, modifies train.py, runs the process using prepare.py, measures the outcome, and decides whether to keep or revert the change.

It's systematic. It's measurable. It's autonomous.

Honest Limitations & Risks

I'd be doing you a disservice if I didn't talk about what doesn't work here. Autoresearch isn't a silver bullet. Here's what keeps me up at night:

1. Not Everything Is Measurable

Team morale. Trust. Psychological safety. Culture. These matter enormously, and they're not easily quantified. You can't just optimize for velocity if it burns out your team. You can't just optimize for email open rates if it damages relationships.

Human judgment still matters. Autoresearch optimizes the measurable. But some of the most important parts of program management are qualitative.

2. The Metric Trap (Goodhart's Law)

Goodhart's Law: "When a measure becomes a target, it ceases to be a good measure."

If you optimize email subject lines for open rates, you might end up with clickbait. If you optimize meeting duration without tracking decision quality, you might end up with fast meetings that don't actually resolve anything.

Your metric must capture what you actually care about. Not just what's easy to measure.

3. Optimization Without Wisdom Is Dangerous

An agent can optimize stakeholder communication for engagement. But it can't understand that this particular stakeholder is going through a tough quarter and needs a different tone. It can't read the room.

Autoresearch handles iteration. Humans handle context and strategy.

4. Privacy and Ethics

If you're iterating on real communication with real stakeholders, that has ethical dimensions. Are you telling people they're part of an A/B test? Are you getting consent? Are you handling their data responsibly?

Transparency matters. This isn't a dark pattern. If you're experimenting on communication, be honest about it.

5. Sample Size and Statistical Significance

Some PM processes have small sample sizes. If you only send exec updates to 5 people, your A/B tests aren't going to be statistically significant. You need volume to measure effectively.

Not every process is a good fit for autoresearch. Focus on high-volume, repeatable processes first.

6. Human Judgment for Strategic Pivots

Autoresearch optimizes within a defined space. But sometimes you need to change the space entirely. Sometimes the right move isn't to optimize sprint structure—it's to change your delivery model completely.

The agent optimizes tactics. You still own strategy.

The Vision: PM as Loop Architect

Here's where this goes in five years:

The best program managers won't be the ones who run the best meetings. They'll be the ones who design the best optimization loops.

Your competitive advantage shifts from execution to architecture. From "I know the right way to do this" to "I know what's worth optimizing and how to measure it."

Imagine a PM role where:

  • Your status reports auto-optimize for stakeholder engagement
  • Your meeting structures evolve based on effectiveness data
  • Your risk register learns which patterns actually predict issues
  • Your sprint planning adapts based on team velocity trends
  • Your vendor evaluations improve with each procurement cycle

You don't run these processes anymore. You architect them. You define the metrics, set the constraints, and let the agent iterate toward optimal.

You spend your time on:

  • Strategy: What should we build? What's the roadmap?
  • Relationships: Stakeholder management, team dynamics, cross-functional alignment
  • Judgment: The calls that require context, wisdom, and human understanding
  • Loop design: What should we optimize next? What new metrics matter?

The tactical execution—the part that's measurable and iterative—runs autonomously in the background.

"What should I optimize?" becomes more important than "How do I do this?"

This is already happening in pockets. I'm building a multi-agent AI fleet (four agents across three machines) that handles content creation, code building, and operations. One agent runs research pipelines overnight. Another manages infrastructure. They work while I sleep.

The same pattern applies to program management. Not because it's trendy. Because it's measurable, and measurement enables autonomous iteration.

Where to Start (Monday Morning Action Plan)

If you're reading this and thinking "I need to try this," here's what I'd do:

1. Pick one high-volume, measurable process. Don't start with your most critical process. Start with something repeatable where you can measure outcomes. Stakeholder emails are a good choice—high volume, clear engagement metrics, low risk.

2. Define your metric. Not "make it better." A number. Open rate, response rate, time-to-response, action completion rate—something you can track week over week.

3. Write your program.md. Even if you don't have autoresearch tooling yet, writing it clarifies your thinking. What's the goal? What's the metric? What are you allowed to change? What are the constraints?

4. Run manual experiments. You don't need AI to start iterating. Try two variations. Measure outcomes. Keep what works. You're building the muscle memory for metric-driven iteration.

5. Instrument your processes. Start tracking the data you'll need. Email analytics, meeting durations, action item completion, velocity trends. You can't optimize what you don't measure.

6. Think in loops. Every time you run a process, ask: "How would I measure if this worked? What would I change next time?" You're training yourself to think like a loop architect.

The tooling will come. I'm building it. Others are too. But the mindset shift can happen now.

Start thinking of your PM work as a set of optimization loops. Define the metrics. Instrument the processes. Iterate based on evidence, not intuition.

That's autoresearch. And it changes everything.
